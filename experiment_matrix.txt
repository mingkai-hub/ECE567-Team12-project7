Why These Settings
- We force discrete actions so mappo, ippo, qmix, and vdn are compared on the same action space.
- We keep the algorithm-specific hyperparameters at BenchMARL defaults to avoid biasing the comparison by hand-tuning one method more than another.
- We reduce max_n_frames from the default 3000000 to 2000000 so the 24 Phase 1 runs stay manageable on Great Lakes while still giving the runs enough time to separate.
- We disable render and wandb-style logging to keep cluster runs lighter and more stable.

MAPPO Ablation Plan

Purpose
- After finishing the 24 Phase 1 runs, run targeted ablations on MAPPO only.
- The goal is to study which MAPPO/PPO training settings most affect performance and stability.
- This is meant to mirror the core message of 2103.01955: MAPPO performance is sensitive to implementation and update choices.

Recommended Ablation Task
- Primary task: pettingzoo/simple_spread
- Optional validation task after that: pettingzoo/simple_reference

Important Note
- The Phase 1 comparison mappo vs ippo already acts as a centralized-critic ablation:
  - mappo = centralized critic
  - ippo = independent critic
- So the MAPPO-only ablation should focus on PPO update hyperparameters rather than repeating that comparison.

MAPPO Baseline For Ablations
- algorithm=mappo
- task=pettingzoo/simple_spread
- seeds=0,1,2
- Keep all Phase 1 fixed parameters unchanged unless explicitly varied below

Ablation Group A: PPO Clip Range
- Parameter: algorithm.clip_epsilon
- Values:
  - 0.05
  - 0.1
  - 0.2
- Reason:
  - Tests how sensitive MAPPO is to PPO clipping aggressiveness
  - This is one of the most important factors highlighted by the paper

Ablation Group B: Data Reuse / On-Policy Update Intensity
- Parameter: experiment.on_policy_n_minibatch_iters
- Values:
  - 15
  - 45
  - 90
- Reason:
  - Tests whether reusing the same collected data too much hurts performance
  - This corresponds closely to the paper's discussion of training epochs and non-stationarity

Ablation Group C: Minibatch Size
- Parameter: experiment.on_policy_minibatch_size
- Values:
  - 200
  - 400
  - 800
- Reason:
  - Tests whether smaller or larger PPO minibatches produce more stable updates

Ablation Group D: Frames Per Batch
- Parameter: experiment.on_policy_collected_frames_per_batch
- Values:
  - 3000
  - 6000
  - 12000
- Reason:
  - Tests whether larger rollout batches improve stability and sample efficiency

Recommended Ablation Order
1. Baseline MAPPO on simple_spread, seeds 0,1,2
2. Clip epsilon ablation
3. on_policy_n_minibatch_iters ablation
4. on_policy_minibatch_size ablation
5. on_policy_collected_frames_per_batch ablation
6. If results are clean and time permits, repeat the most important ablation on simple_reference

Suggested Analysis Questions
- Does smaller clip_epsilon help MAPPO stability?
- Does heavier data reuse hurt MAPPO in cooperative MARL?
- Does larger rollout batch size help MAPPO learn more smoothly?
- Which hyperparameter causes the largest drop from the baseline?

Approximate MAPPO Ablation Size
- Reused from Phase 1:
  - mappo baseline on simple_spread with seeds 0,1,2
  - default-setting runs already covered by Phase 1
